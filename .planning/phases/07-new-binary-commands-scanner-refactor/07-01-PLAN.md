  <name>FilterBlockedTasks — dependency failure propagation</name>
  <files>internal/executor/waves.go, internal/executor/waves_test.go</files>
  <behavior>
    - Test 1 (Empty failedIDs): FilterBlockedTasks(tasks=[T1,T2,T3], failedIDs=[]) returns all 3 tasks unchanged
    - Test 2 (Direct dependency): FilterBlockedTasks(tasks=[T1,T2(depends T1),T3], failedIDs=[1]) returns only [T3] (T2 blocked by T1)
    - Test 3 (Transitive propagation): FilterBlockedTasks(tasks=[T1,T2(depends T1),T3(depends T2),T4], failedIDs=[1]) returns only [T4] (T2 blocked by T1, T3 blocked by T2 transitively)
    - Test 4 (Multiple failures): FilterBlockedTasks(tasks=[T1,T2,T3(depends T1),T4(depends T2)], failedIDs=[1,2]) returns empty (all downstream blocked)
    - Test 5 (Failed task itself excluded): FilterBlockedTasks should NOT include the failed tasks themselves in the returned list — only non-blocked, non-failed tasks are returned
    - Test 6 (No dependencies at all): FilterBlockedTasks(tasks=[T1,T2,T3] all independent, failedIDs=[1]) returns [T2,T3]
  </behavior>
  <implementation>
    Add `FilterBlockedTasks(tasks []TaskItem, failedIDs []int) []TaskItem` to `internal/executor/waves.go`.

    Algorithm (per D-13, reusing adj graph pattern from BuildWaveGroups):
    1. Build adjacency map: for each task, map task.ID -> list of IDs that depend on it (same as BuildWaveGroups's `adj` map)
    2. Initialize blockedSet from failedIDs
    3. BFS from each failed ID through adj: for each blocked task, add its downstream dependents to blockedSet and queue
    4. Return filtered slice: tasks where task.ID is NOT in blockedSet

    CRITICAL (Pitfall 6 from RESEARCH.md): BFS must be fully transitive — use a queue, not single-level iteration. T1 fail -> T2 depends T1 -> T3 depends T2 must ALL be blocked.

    The function signature `FilterBlockedTasks(tasks []TaskItem, failedIDs []int) []TaskItem` is locked per D-13.
  </implementation>
</feature>
