    **Step 1: Get Execution Context** (same as before):

    **Step 2: Decide Execution Mode** (NEW — per D-03):
    - If `has_parallel_opportunity` is false: use sequential mode, skip asking
    - If `auto_mode` is true: use wave mode if `wave_groups` has any wave with 2+ tasks, else sequential
    - Otherwise: ask user "Choose execution mode:" with two options:
      - "Sequential (safe, one task at a time)"
      - "Wave parallel (N tasks in M waves)" — show actual counts from wave_groups

    **Step 3A: Sequential Mode** (simplified from old version):
    - Single Task call to mysd-executor with all pending_tasks
    - No worktree, no wave logic

    **Step 3B: Wave Mode** (NEW — core of Phase 6):
    ```
    For each wave_index, wave in wave_groups:
      Print: "Wave {wave_index+1}/{total_waves}: T{ids} executing in parallel..."

      # Create worktrees for this wave
      For each task in wave:
        Run: mysd worktree create {task.id} "{task.name}"
        Parse JSON output for path and branch

      # Spawn parallel executors (one Task call per task)
      For each task in wave:
        Task: mysd-executor with:
          - change_name, must_items, should_items, may_items, tdd_mode, atomic_commits
          - execution_mode: "wave"
          - assigned_task: {task}
          - worktree_path: {path from create}
          - branch: {branch from create}
          - isolation: "worktree"

      Wait for ALL executors to complete (do NOT abort on single failure — FEXEC-09)

      # Merge loop — ascending task ID order (FEXEC-06)
      Sort completed tasks by ID ascending
      For each task in ascending ID order:
        If task succeeded:
          Run: git merge --no-ff {branch} (in repo root, not worktree)
          If merge conflict:
            Retry up to 3 times (FEXEC-07, per D-05):
              1. Resolve conflict markers in conflicting files
              2. git add resolved files
              3. git commit (complete the merge)
              4. Run: go build ./... && go test ./...
              5. If build/test fails: git merge --abort, retry from step 1
            If still failed after 3 attempts:
              Print error: "Merge failed for T{id}: {reason}"
              Print: "Worktree preserved at {worktree_path} on branch {branch}"
              Print: "To resolve manually: cd {worktree_path}"
              Skip cleanup for this task (FEXEC-08 — failed worktrees preserved)
              Continue to next task
          If merge succeeded:
            Run: mysd worktree remove {task.id} "{branch}"
            Print: "T{id} merged and cleaned up"
        If task failed:
          Print: "T{id} failed — worktree preserved at {worktree_path}"

      Print: "Wave {wave_index+1} complete: {succeeded} succeeded, {failed} failed"

      # CRITICAL: Do NOT create next wave's worktrees until ALL merges are done
      # (per Anti-Pattern 3 from RESEARCH.md — prevents HEAD divergence)
    ```

    **Step 4: Post-Execution Summary** (enhanced):
    - Total tasks completed / failed
    - Preserved worktrees (if any) with paths
    - Reminder: "Run `mysd status` to check overall progress"

    **Key constraints to encode in the SKILL.md text:**
    - Worktrees for wave N+1 MUST NOT be created until wave N's merge loop is fully complete
    - Merge order is always ascending task ID (deterministic)
    - continue-on-failure: one task failing does not abort the wave or merge of other tasks
    - Failed worktrees are preserved, successful ones are deleted
