    ```yaml
    version: 2
    project_name: mysd

    before:
      hooks:
        - go mod tidy

    builds:
      - id: mysd
        main: .
        binary: mysd
        env:
          - CGO_ENABLED=0
        ldflags:
          - -s -w
          - -X main.version={{.Version}}
          - -X main.commit={{.Commit}}
          - -X main.date={{.Date}}
        goos:
          - linux
          - darwin
          - windows
        goarch:
          - amd64
          - arm64
        ignore:
          - goos: windows
            goarch: arm64

    archives:
      - id: mysd
        format: tar.gz
        format_overrides:
          - goos: windows
            format: zip
        name_template: "{{ .ProjectName }}_{{ .Version }}_{{ .Os }}_{{ .Arch }}"

    checksum:
      name_template: "checksums.txt"
      algorithm: sha256

    changelog:
      sort: asc
      filters:
        exclude:
          - "^docs:"
          - "^test:"
          - "^chore:"
    ```

    Note: `release.github` section is intentionally omitted -- GoReleaser auto-detects from git remote.

    ```json

            ]
          }
        ]
      }
    }
    ```

    **Part A: mysd-scan.md SKILL.md** -- Create in both `.claude/commands/` and `plugin/commands/` (identical content).

    Follows the mysd-verify.md pattern: context-only -> present to user -> invoke agent -> user confirms.

    Frontmatter:
    - model: claude-sonnet-4-5
    - description: Scan existing codebase and generate OpenSpec-format spec documents for discovered packages.
    - allowed-tools: Bash, Read, Write, Task

    Content structure:
    - Step 1: Run `mysd scan --context-only` (optionally with `--exclude vendor,testdata`). Parse JSON output containing root_dir, packages array (name, dir, go_files, test_files, has_spec), existing_specs, excluded_dirs, total_files.
    - Step 2 (per D-02): Present scan results to user showing packages to analyze (has_spec=false) and packages skipped (has_spec=true). Ask for confirmation: "Proceed with spec generation for the listed packages? Type 'yes' to proceed, or list package names to exclude."
    - Step 3: For EACH confirmed package, invoke scanner agent via Task tool: `Task: Generate OpenSpec spec for package {name}` with `Agent: mysd-scanner` and context including package name, directory, go_files, test_files, specs directory.
    - Step 4: Report results showing generated specs count, file paths, and next steps (/mysd:spec, /mysd:ff).

    **Part B: mysd-scanner.md agent** -- Create in both `.claude/agents/` and `plugin/agents/` (identical content).

    Follows the mysd-verifier.md pattern: reads source code, generates structured output.

    Frontmatter:
    - model: claude-sonnet-4-5
    - description: Scanner agent. Analyzes Go package source code and generates OpenSpec-format spec documents.
    - allowed-tools: Read, Bash, Write, Glob, Grep

    Content structure:
    - Input section: Receives context about a single Go package (name, directory, go_files, test_files, specs_dir).
    - Process section:
      1. Read Package Source -- read ALL Go source files. Understand exported functions/types/interfaces, behaviors, dependencies, error handling, documentation. Also read test files.
      2. Identify Capabilities -- group into MUST (core functionality), SHOULD (quality attributes), MAY (optional features). Use RFC 2119 keywords.
      3. Generate Spec Files -- create `{specs_dir}/changes/{package_name}/proposal.md` with frontmatter (spec-version: "1.0", title, status: proposed, created date) and body (Summary, Motivation). Create `{specs_dir}/changes/{package_name}/specs/spec.md` with requirements list using MUST/SHOULD/MAY keywords.
      4. Granularity Rules (per D-04) -- one spec per Go package (not per file). Sub-packages get own specs. Skip test-only packages. Skip main packages (cmd/ directories).
    - Output section: Write files to filesystem, report created files.

    Write identical content to both locations (`.claude/commands/` for dev use, `plugin/commands/` for distribution).
  </action>
